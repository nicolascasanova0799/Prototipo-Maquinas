# Requerimientos de producto

Priorización usando **MoSCoW**: **M**ust have · **S**hould have · **C**ould have · **W**on't have (esta fase).

La priorización es una **propuesta de trabajo** basada en la lógica del flujo de negocio (qué es indispensable para el ciclo mínimo Mandante↔Gestor↔Cliente) y debe validarse con el equipo de negocio.

## 1. Transversal / Autenticación

| ID     | Funcionalidad               | Descripción                                                                                                                                                                                    | Prioridad |
| ------ | --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |
| AUTH-1 | Login de usuario            | Acceso diferenciado por perfil: Mandante, Gestor. **(supuesto: Cliente Final y Terreno no requieren login en esta fase)**                                                                      | Must      |
| AUTH-2 | Gestión de roles y permisos | Perfiles con distintos niveles de acceso dentro de cada panel. Se expone en la sección **Configuración** del sidebar (Usuarios y Roles) en ambos paneles _(alineado con prototipo 03/07/2026)_ | Should    |

### 2.2 Gestión de Equipos

| ID     | Funcionalidad                       | Descripción                                                                                                                                                                                                                                       | Prioridad |
| ------ | ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |
| MAN-5  | Asignación de equipos a Gestor      | Selección de **gestor**, **sucursal de destino** (obligatoria, poblada dinámicamente desde las direcciones del gestor) y lote de máquinas a asignar (ver flujo en [diagramas-del-sistema.md](diagramas-del-sistema.md "mention")). El botón "Nueva asignación" pertenece a este módulo, no a Autorización de Movimientos. _(Actualizado 13/07/2026 — se agrega selector obligatorio de sucursal de destino.)_                           | Must      |
| MAN-5b | Guía de Despacho Mandante → Gestor  | Al confirmar el envío de una asignación, el modal ofrece 3 opciones para la GD: **Generar vía API** (simulación 3 pasos: solicitud, procesamiento, respuesta con PDF), **Subir PDF** (carga manual de documento externo) o **Hacer más tarde** (gestionar posteriormente desde la tabla de Asignaciones o el módulo de Guías de Despacho). Queda pendiente validar contrato técnico y manejo de errores de la API. _(Actualizado 13/07/2026 — GD opcional durante o después de la confirmación.)_ | Must      |
| MAN-6  | Autorización de Movimientos         | Bandeja de solicitudes de baja/cambio/movimiento hechas por el Gestor, con aprobación/rechazo del Mandante según roles aprobadores y voto vinculante definidos en los motivos de movimiento.                                                       | Must      |
| MAN-7  | Control de Inventario               | Vista de solo lectura de los inventarios físicos gestionados por cada Gestor: solicitudes en curso, finalizados, resultados y discrepancias detectadas (equipo vs. sistema). El Mandante consulta, no genera ni ajusta inventarios — esa operación es del Gestor. | Should    |
| MAN-8  | Trazabilidad de equipo              | Historial completo de estados y ubicaciones de un equipo, con buscador predictivo por N° de serie, modelo, marca o tipo de máquina.                                                                                                               | Should    |

> **Nota de alcance — Inventario (actualizada 17/07/2026):** el proceso de inventario tiene dos capas: (1) **gestión web** de solicitudes, consultas y ajustes (MAN-7, DIS-5 a DIS-7) y (2) **auditoría física en terreno** mediante la aplicación móvil complementaria. La capa móvil permite validar existencia, condición, evidencia y, si se valida, ubicación o lectura de código. El prototipo web no ejecuta esa auditoría; debe registrar y consultar sus resultados cuando lleguen desde la aplicación móvil.

## 5. Explícitamente fuera de alcance (Won't have — esta fase)

* **Implementación de la aplicación móvil de terreno** (captura de cámara, audio, GPS, QR/código de barras y sincronización real): se gestiona en el proyecto complementario **Prototipo Móvil Máquinas**. El prototipo web conserva solo la consulta y seguimiento de sus resultados.
* **Backend funcional real / lógica de negocio ejecutable** — el entregable de esta fase es una maqueta navegable no funcional.
* **Conectores API reales con SAP/Odoo/ERP Isstec** — solo se documenta el diseño conceptual de integración (ver [arquitectura-de-alto-nivel.md](arquitectura-de-alto-nivel.md "mention")).
* **Gestión de máquinas propias del Gestor** (equipos no pertenecientes al Mandante) — confirmado explícitamente como fuera de alcance en reunión de validación 03/07/2026 (RN-10). Todo equipo gestionado en el maestro pertenece a Carozzi (Mandante).
